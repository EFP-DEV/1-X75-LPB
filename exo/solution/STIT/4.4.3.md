# Exercice 4.4 — Version 3 (résultat stocké + interaction minimale)

## Introduction

Dans cette version, on applique une règle simple :

* **Le traitement décide un résultat** (niveau “stockage”)
* **L’interaction exprime seulement le résultat final** (niveau “Ecrire/Lire”)

Objectif : pouvoir changer l’interface (console, HTML, API…) **sans changer** les règles.

---

## Règles exactes (réponse reçue)

* Draxx : **1835 compris** → exclusion si `age >= 1835`
* Sklounst : **18 compris**, **35 non compris** → exclusion si `age >= 18` et `age < 35`
* Tous les autres : autorisé

---

## Pourquoi c’est important

### 1) Deux raisons de changer différentes

* **Traitement** : change si la règle change (rare, mais important)
* **Interaction** : change si le support change (souvent : console → HTML → API → mobile…)

Si on mélange les deux, chaque modification d’interface oblige à retoucher la logique → risque d’erreurs.

### 2) Même traitement, plusieurs interfaces

Le traitement produit une information stable (“peut voter / ne peut pas voter”).
Ensuite, chaque interface peut l’exprimer autrement :

* console : `Ecrire message`
* HTML : afficher un `<p>`
* API : renvoyer du JSON
* etc.

Le cœur ne bouge pas.

### 3) Préparation aux fonctions (return)

Aujourd’hui : on stocke le résultat dans `message`.
Plus tard : une fonction renverra ce résultat (même logique, autre forme).

## Inspection des valeurs (bornes)

### Draxx

* `age = 1835` → `age >= 1835` vrai → **Ne peut pas voter**
* `age = 1834` → faux → **Peut voter**

### Sklounst

* `age = 18` → `>= 18` vrai et `< 35` vrai → **Ne peut pas voter**
* `age = 34` → vrai et vrai → **Ne peut pas voter**
* `age = 35` → `< 35` faux → **Peut voter**

---

## Conclusion

Version plus “propre” :

* le programme calcule un **résultat** (stocké dans `message`)
* l’interface ne fait que **l’exprimer** à la fin

Résultat : si demain on remplace la console par une page HTML ou une API, on garde le même traitement.
